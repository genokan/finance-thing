# UI review — clarity, consistency, usability

Date: 2026-07-03. Method: full read of every page component + stylesheet, then a
live Playwright walkthrough of all 11 pages as a seeded test user — clicking
buttons, opening modals, pressing keys, and screenshotting each page.

Two deliverables came out of it: the findings below, and a restructuring of the
CSS into a **portable library** (`src/ui/glass.css`) with finance-thing as its
first consumer (see "CSS library" at the end).

## Findings

### Correctness / data consistency

1. **Budgets used a different income basis than the dashboard.** The Budgets
   header showed $12,083/mo (gross ÷ 12) while the dashboard's 50/30/20 uses
   net take-home ($8,551). The same Needs spending read 17.4% on one page and
   31.7% on the other. → Budgets now uses the same net-monthly figure via
   `estimateTax` (50/30/20 is conventionally computed on take-home).
2. **`money(-0)` rendered "-$0".** Visible as a red "-$0" debt tile for users
   with no debt. → format guard added; zero is never signed.
3. **Zero styled as alarming.** Accounts header showed "$0 owed" in red.
   → negative tone only applies to non-zero amounts.

### Consistency

4. **The bucket concept had two color systems.** Outflow shows Needs=gold /
   Wants=indigo / Savings=mint (`BucketBadge`); Budgets had its own map with
   ESSENTIAL=neutral, DISCRETIONARY=gold — the same idea colored differently
   page to page, in SHOUTING enum case. → `BucketBadge` is now the single
   source; Budgets uses it.
5. **Dashboard/Budgets 50/30/20 detail bars didn't match their own legend.**
   Meter and legend say gold/indigo/mint; the bars beneath rendered
   gold/gold/mint. → bars now take the bucket color (`info` tone added).
6. **Copy drift:** History's empty state said to use "Record this month" but
   the dashboard button is labeled "Snapshot Jul 2026". → copy now names the
   button as rendered.
7. **~60 ad-hoc inline `style={{…}}`** for spacing/width/alignment repeated
   across pages (`marginTop:-4` hacks, `display:flex;gap:8` wrappers,
   `fontSize:13` notes). → replaced with a small set of library utilities
   (`.cluster`, `.note`, `.w-narrow`, `.btn.block`, `.success-text`).

### Usability

8. **Escape did not close modals** (verified live), no focus trap, background
   scrolled behind the dialog. → `Modal` now renders a native `<dialog>` via
   `showModal()`: Esc-to-close, focus containment, `::backdrop`, and scroll
   lock come from the platform, not JS.
9. **Every delete was a single unconfirmed click.** Accounts, holdings, debts,
   expenses, contributions, categories, institutions, users — one mis-tap on
   ✕ soft-deletes with no undo. → two-step `ConfirmButton` (click ✕ → becomes
   "Sure?" for a beat) used everywhere destructive.
10. **Icon buttons had no accessible names** (bare ✎ / ✕ glyphs, tiny hit
    areas). → aria-labels everywhere, minimum hit target enforced in CSS.
11. **Checkboxes were browser-default blue** amid a gold design system.
    → `accent-color` token applied to checkbox/radio/range.
12. **Bare "Loading…" text** on every page load; avatar renders "?" while
    settings load. → CSS-only spinner with label (reduced-motion falls back
    to text).
13. **Debt page burned ~300px on empty sections** ("No 0% promos", "No
    short-term debt") even when only one group had content. → empty groups are
    hidden; one combined empty state when there are no debts at all.
14. **Truncated placeholder** "blank = amortized min" clipped in the account
    modal. → shortened; the label carries the explanation.
15. **Stale-session edge:** a JWT for a deleted user still renders the app
    shell with all-$0 data and console errors (found via an orphaned refresh
    cookie). Not fixed here — noted for a server-side pass (refresh should 401
    when the user row is gone).

### Quality floor (previously missing, now in the library)

- `:focus-visible` rings on all interactive elements (was: inputs only)
- `prefers-reduced-motion` respected for every animation/transition
- Styled thin scrollbars; custom select chevron (replaces default arrow)
- Consistent `.note` / `.success-text` / `.error-text` message styles

## CSS library — `src/ui/glass.css`

The stylesheet is now split for portability:

- **`src/ui/glass.css`** — the reusable library ("Glass UI"). Zero
  app-specific selectors. Structured with `@layer tokens, base, components,
  utilities` so consumers can override anything without specificity fights.
  Theming is entirely token-driven: swap the `--accent*` values and the whole
  system (buttons, focus rings, seg controls, badges) follows, including
  states derived with `color-mix()`.
  - *tokens*: palette, glass surfaces, type roles, radii, shadows, motion
  - *base*: reset, focus-visible, reduced-motion, form-control theming,
    scrollbars, selection, `.glass-scene` (opt-in ambient background + grain)
  - *components*: card, stat, hero, list/row, amt-cell, btn (solid / ghost /
    danger / sm / block), iconbtn, field/input/field-row, seg, badge, chip,
    meter, bar, legend, dialog (native `<dialog>`), knob/slider, empty,
    loading, toast, dropdown menu
  - *utilities*: `.cluster`, `.stack`, `.note`, `.success-text`,
    `.error-text`, `.w-narrow`, text tones (`.pos .neg .dim .accent .num`)
- **`src/app.css`** — finance-thing specifics: brand lockup, top/bottom nav,
  user menu, auth screen, chart paddings, page scaffolding.

To reuse in another project: copy `glass.css`, add `class="glass-scene"` to
`<body>` (or skip it for a plain dark surface), override tokens in your own
`@layer tokens` block, and use the same class vocabulary.
